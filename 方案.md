# OpenSpec + Superpowers + 多 Agent 自主协作 Harness 工程方案

## 0. 方案结论

本方案采用四层架构：

- **OpenSpec**：工程事实库，沉淀需求、规范、变更、设计、任务和归档后的 source of truth。
- **Superpowers**：Agent 行为纪律层，约束每个 Agent 先澄清、先设计、写计划、测试驱动、审查和验证。
- **auok CLI**：执行与验证层，负责场景运行、评分、报告、回归门禁，并提供类 `opsx` 命令。
- **Multi-Agent Orchestrator**：自主协调层，负责任务分派、状态推进、证据收集、失败重试和最终交付候选。

目标不是让一个 Agent 从头猜到尾，而是让多个 Agent 基于共享文件和确定性命令协作：

```text
OpenSpec = 多 Agent 的共享事实库
Superpowers = 每个 Agent 的行为纪律
auok CLI = 自动验证与门禁
Orchestrator = 多 Agent 状态机和协调器
```

推荐自动化边界：

- Agent 可以自动完成需求整理、OpenSpec change、设计、任务拆分、实现、测试、评审、修复、报告和归档候选。
- 人只保留关键审批：目标确认、风险边界确认、降低门禁阈值、生产配置修改、最终 merge / archive / release。

## 1. 建设目标

建设一个 AI/Agent harness 工程，用于：

- 管理评测能力规范。
- 管理测试场景和评测数据。
- 自动执行被测 Agent / 模型 / API。
- 自动评分、生成报告、执行 CI 门禁。
- 通过多个 Agent 自主推进变更。
- 将所有关键过程沉淀为可审查文件，而不是散落在聊天上下文中。

最终开发体验：

```bash
auok auto "新增工具调用评测能力，覆盖天气查询必须调用 weather 工具"
```

系统自动完成：

```text
创建 OpenSpec change
-> 生成 proposal/design/tasks/spec delta
-> 分派 Dev/QA/Review Agent
-> 实现 scenario/runner/grader/report
-> 跑测试和 auok gate
-> 审查并自动修复
-> 生成交付报告
-> 等待人工确认 archive / merge
```

## 3. 多 Agent 角色设计

### 3.1 Orchestrator Agent

职责：

- 接收目标。
- 创建或选择 workflow。
- 创建状态文件。
- 分派任务给其他 Agent。
- 收集 handoff 和 evidence。
- 根据 gate 结果决定推进、返工或等待人工确认。

不可做：

- 不直接修改业务实现。
- 不跳过失败 gate。
- 不自行降低验收标准。

输出：

```json
{
  "agent": "orchestrator",
  "change": "add-tool-call-eval",
  "state": "qa_failed",
  "next": "dev_fix",
  "reason": "auok gate failed: pass_rate=0.91 < 1.0"
}
```

### 3.2 Spec Agent

职责：

- 阅读 `openspec/specs/`。
- 创建 `openspec/changes/<change-id>/`。
- 生成 `proposal.md`、`design.md`、`tasks.md`、`specs/*/spec.md`。
- 明确不做什么、验收标准、风险和影响面。

Superpowers 使用重点：

- `brainstorming`
- `writing-plans`

输出：

```json
{
  "agent": "spec",
  "status": "done",
  "files": [
    "openspec/changes/add-tool-call-eval/proposal.md",
    "openspec/changes/add-tool-call-eval/design.md",
    "openspec/changes/add-tool-call-eval/tasks.md"
  ],
  "next": "dev"
}
```

### 3.3 Dev Agent

职责：

- 按 `tasks.md` 实现代码。
- 优先补测试或 scenario。
- 实现 adapter、runner、grader、report、CLI。
- 更新任务状态。

Superpowers 使用重点：

- `test-driven-development`
- `executing-plans`
- `systematic-debugging`

不可做：

- 不改需求。
- 不降低 gate。
- 不删除无关代码。

### 3.4 QA Agent

职责：

- 设计 scenario。
- 校验 scenario schema。
- 执行 `auok run`、`auok grade`、`auok gate`。
- 生成测试报告。
- 把失败转成可执行反馈。

输出：

```json
{
  "agent": "qa",
  "status": "failed",
  "evidence": [
    "runs/add-tool-call-eval/report.md",
    "auok gate failed: critical scenario tool-call.weather.basic failed"
  ],
  "next": "dev_fix"
}
```

### 3.5 Review Agent

职责：

- 审查 spec 是否覆盖需求。
- 审查代码是否符合 design/tasks。
- 审查测试是否能证明行为。
- 审查是否存在过度设计、未验证声明、无关改动。

Superpowers 使用重点：

- `requesting-code-review`
- `receiving-code-review`

输出要求：

- findings 优先，按严重程度排序。
- 每个问题必须指向文件和证据。
- 没有问题时明确说明残余风险。

### 3.6 Archive Agent

职责：

- 确认所有 gate 通过。
- 确认 tasks 完成。
- 准备 `openspec archive`。
- 生成归档候选报告。

不可做：

- 不在有失败 gate 时归档。
- 不自动合入主分支。
- 不自动发布。

## 4. 状态机设计

状态文件放在：

```text
agent-orchestration/states/<change-id>.yaml
```

示例：

```yaml
change: add-tool-call-eval
goal: 新增工具调用评测能力，覆盖天气查询必须调用 weather 工具
state: implementing
owner: orchestrator

agents:
  spec:
    status: done
    last_output: agent-orchestration/handoffs/add-tool-call-eval/spec-to-dev.json
  dev:
    status: running
  qa:
    status: pending
  review:
    status: pending
  archive:
    status: pending

gates:
  openspec_validate:
    status: pass
    command: openspec validate add-tool-call-eval --strict
  unit_tests:
    status: pending
    command: npm test
  auok_gate:
    status: pending
    command: auok gate runs/add-tool-call-eval --min-pass-rate 1.0 --no-critical-failures
  review:
    status: pending

human_approval:
  required_for:
    - merge
    - archive
    - release
    - lowering_gate_threshold
    - production_config_change
  status: not_requested
```

`human_approval.status` 取值：

```text
not_requested  # 尚未请求人工确认
requested      # Archive Agent 已生成证据包，等待确认
approved       # 人已确认，可以继续 archive/merge/release
rejected       # 人拒绝，状态机回到 blocked 或 fixing
```

审批证据包必须包含：

- change id。
- proposal / design / tasks 路径。
- OpenSpec validate 结果。
- 单测或项目测试结果。
- `auok gate` 结果。
- Review Agent findings。
- Archive Agent 建议动作。

核心状态：

```text
created
-> spec_planning
-> spec_review
-> implementing
-> qa_running
-> review_running
-> fixing
-> ready_for_archive
-> waiting_human_approval
-> archived
```

失败分支：

```text
qa_failed -> fixing
review_failed -> fixing
openspec_invalid -> spec_planning
blocked -> waiting_human_input
```

`blocked` 的进入条件：

- 自动重试超过上限。
- 需求或验收标准冲突。
- 需要降低 gate 阈值。
- 需要生产配置或敏感权限。
- Agent 无法从现有证据判断下一步。

## 5. Workflow 定义

Workflow 文件放在：

```text
agent-orchestration/workflows/change-lifecycle.yaml
```

示例：

```yaml
name: change-lifecycle
description: 多 Agent 自主完成 OpenSpec change 到归档候选

states:
  created:
    next: spec_planning

  spec_planning:
    agent: spec
    inputs:
      - openspec/specs
      - user_goal
    outputs:
      - openspec/changes/{change}/proposal.md
      - openspec/changes/{change}/design.md
      - openspec/changes/{change}/tasks.md
    next: spec_review

  spec_review:
    agent: review
    command: openspec validate {change} --strict
    on_pass: implementing
    on_fail: spec_planning

  implementing:
    agent: dev
    inputs:
      - openspec/changes/{change}/tasks.md
    next: qa_running

  qa_running:
    agent: qa
    commands:
      - auok run --change {change} --out runs/{change}
      - auok grade runs/{change}
      - auok gate runs/{change} --min-pass-rate 1.0 --no-critical-failures
    on_pass: review_running
    on_fail: fixing

  review_running:
    agent: review
    on_pass: ready_for_archive
    on_fail: fixing

  fixing:
    agent: dev
    next: qa_running

  ready_for_archive:
    agent: archive
    requires_human_approval: true
    next: waiting_human_approval
```

### 5.1 多 Agent 运行时

多 Agent 有两种落地模式，MVP 先做第一种：

| 模式 | 说明 | 适用阶段 |
| --- | --- | --- |
| 文件状态机模式 | `auok` 只创建状态文件、handoff 和下一步指令，由当前 Agent 或人工触发下一步 | MVP |
| Agent Runtime 模式 | `auok` 通过可配置 executor 调用 Codex/Claude/其他 Agent CLI 或平台 API | 后续增强 |

MVP 不要求 `auok` 真正启动多个独立进程。它必须先保证：

- 每个 Agent 的输入输出文件明确。
- 状态推进可恢复。
- 失败可返工。
- 所有完成声明有命令证据。

后续可在 `agent-orchestration/config.yaml` 中配置 executor：

```yaml
executors:
  spec:
    type: codex
    command: codex run --agent spec
  dev:
    type: codex
    command: codex run --agent dev
  qa:
    type: local
    command: auok verify {change}
```

如果运行环境不支持自动拉起子 Agent，`auok agent status` 必须输出下一步可复制命令，而不是假装已经自动执行。

## 6. OpenSpec 使用方式

OpenSpec 是所有 Agent 的共享事实库。

初始化：

```bash
npm install -g @fission-ai/openspec@<pinned-version>
openspec init --tools codex
```

如果项目已经把 OpenSpec 固定在 `devDependencies`，优先使用：

```bash
npx openspec init --tools codex
```

推荐 specs：

```text
openspec/specs/harness-core/spec.md
openspec/specs/scenario-model/spec.md
openspec/specs/runner/spec.md
openspec/specs/adapter/spec.md
openspec/specs/grader/spec.md
openspec/specs/report/spec.md
openspec/specs/ci-gate/spec.md
openspec/specs/agent-orchestration/spec.md
```

每个变更：

```text
openspec/changes/<change-id>/
  proposal.md
  design.md
  tasks.md
  specs/
```

Agent 必跑命令：

```bash
openspec status --change <change-id> --json
openspec validate <change-id> --strict
openspec instructions apply --change <change-id> --json
```

归档候选：

```bash
openspec archive <change-id> --yes
```

注意：`archive` 建议由 Archive Agent 准备，人确认后执行。

## 7. Superpowers 使用方式

Superpowers 约束每个 Agent 的做事方式，不保存长期事实。

安装到 Codex：

```text
Fetch and follow instructions from https://raw.githubusercontent.com/obra/superpowers/refs/heads/main/.codex/INSTALL.md
```

各 Agent 默认行为：

| Agent | 必用 Superpowers 能力 |
| --- | --- |
| Spec Agent | brainstorming、writing-plans |
| Dev Agent | test-driven-development、executing-plans、systematic-debugging |
| QA Agent | verification-before-completion |
| Review Agent | requesting-code-review、receiving-code-review |
| Archive Agent | verification-before-completion |
| Orchestrator | writing-plans、verification-before-completion |

关键规则：

- Superpowers 产生的结论必须写回 OpenSpec 或 handoff 文件。
- 不能只在聊天里完成设计。
- 不能用“我认为通过了”替代命令输出。
- 失败必须转成状态文件和下一步任务。

## 8. auok CLI 设计

`auok` 是本项目提供的统一命令行入口，不是 OpenSpec 或 Superpowers 自带命令。实现时建议在 `package.json` 中声明 bin：

```json
{
  "name": "auok",
  "bin": {
    "auok": "harness/cli/main.js"
  }
}
```

本地开发使用：

```bash
npm link
auok --help
```

CI 或未全局安装时使用项目脚本转发：

```bash
npm run auok -- validate --all
```

`auok` 内部可以调用 OpenSpec CLI，也可以直接读写 `openspec/`、`agent-orchestration/` 和 `runs/`。对用户来说，除初始化和调试外，日常入口统一使用 `auok`。

### 8.1 变更生命周期命令

```bash
auok new <change-id>
auok ff <change-id>
auok status <change-id> --json
auok validate <change-id>
auok apply <change-id>
auok verify <change-id>
auok sync <change-id>
auok archive <change-id>
```

说明：`auok` 的生命周期命令语义参考 OpenSpec OPSX，但终端入口统一采用常规子命令格式，即 `auok <command> ...`。不再保留冒号式或 OPSX 前缀式兼容别名。

命令语义：

| 命令 | 语义 |
| --- | --- |
| `auok new <change-id>` | 创建 `openspec/changes/<change-id>/` 和状态文件 |
| `auok ff <change-id>` | 生成 proposal、design、tasks、spec delta 的缺失项 |
| `auok status <change-id>` | 输出 change、agent、gate 状态 |
| `auok validate <change-id>` | 校验指定 change、scenario schema、agent 状态一致性 |
| `auok validate --all` | 校验全部 OpenSpec、全部 scenario、全部 workflow schema |
| `auok apply <change-id>` | 按 tasks 执行实现，通常由 Dev Agent 使用 |
| `auok verify <change-id>` | 执行 validate、测试、run、grade、gate、review 检查 |
| `auok sync <change-id>` | 将 delta spec 同步到主 specs，但不归档 |
| `auok archive <change-id>` | 归档 change；默认要求人工 approval 状态为 approved |

### 8.2 多 Agent 自动化命令

```bash
auok auto "<goal>"
auok auto --change add-tool-call-eval
auok agent status add-tool-call-eval
auok agent resume add-tool-call-eval
auok agent handoff add-tool-call-eval --from qa --to dev
auok agent approve add-tool-call-eval --action archive
```

命令语义：

| 命令 | 作用 |
| --- | --- |
| `auok auto "<goal>"` | 创建 change 和状态机，启动多 Agent 工作流 |
| `auok agent status` | 查看 Agent 状态、gate 状态、下一步 |
| `auok agent resume` | 从状态文件恢复执行 |
| `auok agent handoff` | 查看或生成 Agent 交接文件 |
| `auok agent approve` | 人工批准 archive/merge/release 等关键动作 |

### 8.3 评测执行命令

```bash
auok list scenarios
auok run --capability tool-calling --adapter mock --out runs/add-tool-call-eval
auok grade runs/add-tool-call-eval
auok report runs/add-tool-call-eval --format md,json
auok compare runs/baseline runs/add-tool-call-eval
auok gate runs/add-tool-call-eval --min-pass-rate 1.0 --no-critical-failures
```

`auok run --change <change-id>` 是自动化快捷入口，语义是：

- 读取 `openspec/changes/<change-id>/tasks.md` 和 spec delta。
- 找到本次变更新增或受影响的 scenarios。
- 如果没有可推导场景，则回退到 `--capability smoke`。
- 输出到 `runs/<change-id>/`。

第一阶段如果不实现变更影响分析，可以先要求显式传入 `--capability` 或 scenario id。

### 8.4 JSON 输出

所有自动化命令必须支持 `--json`：

```json
{

  "state": "qa_failed",
  "agents": {
```

## 9. Handoff 协议

Agent 之间不得只靠对话交接，必须写 handoff 文件。

路径：

```text
agent-orchestration/handoffs/<change-id>/<from>-to-<to>.json
```

格式：

```json
{
  "change": "add-tool-call-eval",
  "from": "qa",
  "to": "dev",
  "status": "failed",
  "summary": "critical scenario failed because no weather tool call was emitted",
  "evidence": [
    "runs/add-tool-call-eval/report.md",
    "runs/add-tool-call-eval/results.jsonl"
  ],
  "required_action": [
    "Inspect adapter output normalization",
    "Fix tool_calls extraction",
    "Re-run auok gate"
  ],
  "commands_to_reproduce": [
    "auok run tool-call.weather.basic --adapter mock --out runs/repro",
    "auok gate runs/repro --min-pass-rate 1.0 --no-critical-failures"
  ]
}
```

## 10. Harness 领域模型

### 10.1 Scenario

```yaml
id: tool-call.weather.basic
title: 查询天气时必须调用天气工具
capability: tool-calling
severity: critical
tags: [tool-call, weather, smoke]
input:
  user: "帮我查一下今天北京天气"
expected:
  tool_calls:
    - name: weather
grader:
  type: rule
  rules:
    - path: "$.tool_calls[*].name"
      contains: weather
```

### 10.2 Adapter

```ts
interface HarnessAdapter {
  name: string
  invoke(input: ScenarioInput, context: RunContext): Promise<AdapterResult>
}
```

第一阶段只做：

- `mock`：保证 harness 自身可测。
- `api`：调用业务 API 或模型 API。

### 10.3 Grader

优先级：

1. `rule`：确定性评分，适合 hard gate。
2. `snapshot`：黄金样本对比，适合结构化报告。
3. `judge`：LLM-as-judge，适合语义质量，不默认作为 hard gate。

### 10.4 Report

```text
runs/<run-id>/
  run.yaml
  results.jsonl
  summary.json
  report.md
```

## 11. 人工介入边界

尽量减少人工介入，但保留必要关口。

Agent 可自动执行：

- 创建 OpenSpec change。
- 生成 proposal/design/tasks/spec delta。
- 实现代码。
- 生成 scenario。
- 跑测试和 gate。
- 审查代码。
- 修复失败。
- 生成交付报告。
- 准备归档候选。

必须人工确认：

- 最终 merge。
- 最终 archive。
- release。
- 降低 gate 阈值。
- 修改生产配置。
- 删除大量文件。
- 启动大规模付费模型评测。
- 接受带残余 critical 风险的变更。

原因：如果 Spec Agent 一开始理解错目标，后续 Agent 可能会高效地实现错误方案。保留最终确认是必要的风险控制。

## 12. CI 门禁

GitHub Actions 示例：

```yaml
name: harness

on:
  pull_request:
  push:
    branches: [main]

jobs:
  harness:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm install
      - run: npm run auok -- validate --all
      - run: npm run auok -- run --capability smoke --adapter mock --out runs/ci
      - run: npm run auok -- grade runs/ci
      - run: npm run auok -- report runs/ci --format md,json
      - run: npm run auok -- gate runs/ci --min-pass-rate 1.0 --no-critical-failures
```

门禁标准：

- OpenSpec 结构合法。
- Scenario schema 合法。
- Smoke 场景全部通过。
- Critical 场景不得失败。
- 报告必须成功生成。
- Review Agent 无 blocking finding。

## 13. MVP 实施计划

### 阶段 1：规范与规则底座

任务：

- 安装 OpenSpec。
- 安装 Superpowers。
- 初始化 `openspec/`。
- 创建初始 specs。
- 创建 `AGENTS.md`。
- 创建 Agent contracts。

验收：

- 每个 Agent 的输入、输出、可写范围明确。

任务：

- 实现 `auok run`。
- 实现 mock adapter。

- `results.jsonl`、`summary.json`、`report.md` 可生成。

### 阶段 3：多 Agent 状态机

任务：

- 实现 `agent-orchestration/workflows/change-lifecycle.yaml`。

## 15. 成功标准

方案落地成功的标志：

- 一个目标可以通过 `auok auto "<goal>"` 自动推进到归档候选。
- OpenSpec change 完整，且能通过 validate。
- 多 Agent 之间通过状态文件和 handoff 文件交接。
- QA 失败和 Review 失败能自动返工。
- 所有完成声明都有命令证据。
- CI 能阻止关键评测失败合入。
- 人只在关键风险点介入。

## 16. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| Spec Agent 理解错需求 | proposal/design 进入人工轻审或 Review Agent 独立审查 |
| Agent 互相覆盖改动 | 每个 Agent 限定可写目录，状态机串行关键写操作 |
| Review Agent 自我确认 | Review Agent 独立上下文，基于 diff、spec 和测试证据审查 |
| LLM judge 波动 | 不作为第一阶段 hard gate |
| 自动化误降门禁 | 降低阈值必须人工确认 |
| runs 数据膨胀 | 只提交 baseline 和摘要，大日志走 artifact |
| 循环修复无法收敛 | 设置最大重试次数，进入 `blocked` 等人工判断 |

建议最大自动重试：

```yaml
retry_policy:
  qa_failed:
    max_attempts: 3
  review_failed:
    max_attempts: 2
  openspec_invalid:
    max_attempts: 2
  on_exceeded: blocked
```

## 17. GitHub 分发与一键安装

后期上传 GitHub 时，建议不要提交 OpenSpec 和 Superpowers 的全局安装产物，也不要提交用户 HOME 目录下的 skill、plugin、symlink。正确方式是：**仓库提交项目配置和 bootstrap 脚本，用户 clone 后一条命令完成安装和初始化**。

### 17.1 应该提交到 GitHub 的内容

```text
scripts/
  bootstrap.sh
  verify-env.sh
  install-openspec.sh
  install-superpowers-codex.sh
  install-superpowers-claude.sh

openspec/
  config.yaml
  specs/
  changes/

agent-orchestration/
  workflows/
  contracts/

superpowers/
  usage.md
  project-rules.md

AGENTS.md
package.json
README.md
```

说明：

- `openspec/specs` 和 `openspec/changes` 是项目事实，应该提交。
- `agent-orchestration/workflows` 和 `contracts` 是多 Agent 协作协议，应该提交。
- `superpowers/usage.md` 只保存本项目如何使用 Superpowers，不复制官方 Superpowers skill 内容。
- `scripts/bootstrap.sh` 负责本机安装和初始化。

### 17.2 不应该提交的内容

```text
node_modules/
runs/* 大体量运行日志
~/.codex/superpowers/
~/.agents/skills/superpowers
全局 npm 安装目录
本机 token / API key / .env
```

这些属于本机环境或敏感数据，应该通过 `.gitignore` 排除。

### 17.3 推荐 package.json scripts

```json
{
  "name": "auok",
  "version": "0.1.0",
  "bin": {
    "auok": "harness/cli/main.js"
  },
  "scripts": {
    "auok": "node harness/cli/main.js",
    "bootstrap": "bash scripts/bootstrap.sh",
    "verify-env": "bash scripts/verify-env.sh",
    "openspec-validate": "openspec validate --all --strict",
    "auok-smoke": "npm run auok -- run --capability smoke --adapter mock --out runs/smoke && npm run auok -- grade runs/smoke && npm run auok -- report runs/smoke && npm run auok -- gate runs/smoke --min-pass-rate 1.0 --no-critical-failures",
    "agent-status": "npm run auok -- agent status"
  },
  "devDependencies": {}
}
```

版本建议：

- MVP 阶段 `auok` 使用零外部依赖，保证无网络也能运行 smoke gate。
- OpenSpec CLI 作为可选外部工具安装；如果后续加入 npm 依赖，发布到 GitHub 前必须固定版本。
- 用 lockfile 保证 CI 和开发机安装一致。
- 全局安装仍可作为开发者便利项，但不要依赖全局安装作为唯一入口。

### 17.4 bootstrap.sh 设计

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT_DIR"

echo "[1/5] Checking Node.js..."
node --version
npm --version

echo "[2/5] Installing npm dependencies..."
npm install

echo "[3/5] Verifying OpenSpec availability..."
if command -v openspec >/dev/null 2>&1; then
  openspec --version
else
  echo "OpenSpec CLI not found. Install it separately when OpenSpec validation is required."
fi

echo "[4/5] Configuring Agent environment..."
case "${AUOK_AGENT_TARGET:-none}" in
  codex)
    bash scripts/install-superpowers-codex.sh
    ;;
  claude)
    bash scripts/install-superpowers-claude.sh
    ;;
  none)
    echo "Skipping Agent-specific Superpowers install."
    ;;
  *)
    echo "Unsupported AUOK_AGENT_TARGET: ${AUOK_AGENT_TARGET}"
    exit 1
    ;;
esac

echo "[5/5] Verifying project..."
npm link || echo "Skipping global auok link; use: npm run auok -- <command>"
npm run verify-env

echo "Bootstrap complete."
```

注意：

- `openspec/` 已作为项目事实提交到仓库，bootstrap 不自动覆盖。
- OpenSpec CLI 缺失时不阻塞本地 auok smoke 验证。
- Superpowers 安装会写用户 HOME 目录，所以必须由脚本在本机执行，不提交结果。
- bootstrap 支持 `--codex`、`--claude`、`--no-agent`，也支持 `AUOK_AGENT_TARGET=codex|claude|none`。

### 17.5 Agent 环境安装脚本

Codex：

```bash
#!/usr/bin/env bash
set -euo pipefail

SUPERPOWERS_DIR="${HOME}/.codex/superpowers"
SKILLS_LINK="${HOME}/.agents/skills/superpowers"

mkdir -p "${HOME}/.codex" "${HOME}/.agents/skills"

if [ ! -d "$SUPERPOWERS_DIR/.git" ]; then
  git clone https://github.com/obra/superpowers.git "$SUPERPOWERS_DIR"
else
  git -C "$SUPERPOWERS_DIR" pull --ff-only
fi

if [ ! -e "$SKILLS_LINK" ]; then
  ln -s "$SUPERPOWERS_DIR/skills" "$SKILLS_LINK"
fi

echo "Superpowers installed for Codex."
echo "Restart Codex so skills are discovered."
```

Claude：

```bash
#!/usr/bin/env bash
set -euo pipefail

SUPERPOWERS_DIR="${HOME}/.claude/superpowers"

mkdir -p "${HOME}/.claude"

if [ ! -d "$SUPERPOWERS_DIR/.git" ]; then
  git clone https://github.com/obra/superpowers.git "$SUPERPOWERS_DIR"
else
  git -C "$SUPERPOWERS_DIR" pull --ff-only
fi

cat > "${HOME}/.claude/AUOK_SUPERPOWERS.md" <<EOF
# auok Superpowers Notes

Superpowers has been cloned to:

${SUPERPOWERS_DIR}

Claude Code integration varies by local setup. Use the project's AGENTS.md,
superpowers/usage.md, and the cloned Superpowers materials as behavior guidance.
EOF
```

如果团队不希望脚本直接联网安装 Superpowers，可以使用：

```bash

```

### 17.8 CI 中的安装方式

CI 不需要安装 Superpowers，因为 CI 只做确定性验证，不跑交互式 Agent。

```yaml
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
  with:
    node-version: 20
- run: npm install
- run: npm run auok -- validate --all
- run: npm run auok-smoke
```

也就是说：

- 开发者本机：`bootstrap` 安装项目依赖，并按选择配置 Codex、Claude 或不配置 Agent 环境。
- CI：只安装 npm 依赖，跑 OpenSpec validate 和 auok gate。
- Agent 环境：Codex 安装后重启 Codex；Claude 安装后按本机 Claude Code 约定读取 `AGENTS.md` 和 `superpowers/usage.md`。
