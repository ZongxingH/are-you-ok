# auok Harness 工程方案

## 0. 方案结论

auok 是一个面向大模型编码环境的 Harness 工程实现。它通过 OpenSpec 和 Superpowers 两类工具，把“规范、纪律、编排、验证、归档”串成一个可执行闭环。当前实现必须严格遵守三层边界：

本方案不是把 OpenSpec、Superpowers、auok 当成三个并列产品，而是：

- **Spec（OpenSpec）= 规范层**：定“做什么”。用结构化文档锁定需求、接口、行为规范、验收标准和归档后的 source of truth。
- **Superpowers = 纪律层**：定“怎么做”。把澄清、设计、计划、TDD、调试、评审、验证等工程流程固化为 Agent 可复用技能和行为约束。
- **Harness（auok）= 协作 / 编排层**：定“谁来做 + 怎么管”。负责 `/auok` 入口、角色分工、任务调度、权限边界、状态机、handoff、验证证据、gate 和最终归档闭环。

底层确定性能力只能做落盘、状态、校验、运行、评分、报告、gate 和归档移动；不能硬编码架构分析、需求决策、实现策略、QA 诊断、Review 判断或归档就绪判断。这些智能判断必须由当前大模型会话中的 Agent + Skill 完成。

实现关系：

```text
auok Harness
-> 使用 OpenSpec 管规范和验收标准
-> 使用 Superpowers 管 Agent 工程纪律
-> 自己负责多 Agent 编排、验证、状态和归档闭环
```

`/auok` slash command、Multi-Agent Orchestrator、状态机、handoff、run/grade/report/gate 都属于 auok Harness 的实现组件。底层 Node 代码只作为 Codex / Claude 中 `/auok` 工作流需要时调用的确定性能力，不作为用户日常入口。

目标不是让一个 Agent 从头猜到尾，而是让当前主 Agent 作为 Orchestrator 拉起多个独立 Agent，并基于共享文件和确定性命令协作。

推荐自动化边界：

- Agent 可以自动完成需求整理、OpenSpec change、设计、任务拆分、实现、测试、评审、修复、报告和归档候选。
- 人只保留关键审批：目标确认、风险边界确认、降低门禁阈值、生产配置修改、最终 merge / archive / release。

## 1. 建设目标

建设一个 AI/Agent harness 工程，用于：

- 管理评测能力规范。
- 管理测试场景和评测数据。
- 自动执行被测 Agent / 模型 / API。
- 自动评分、生成报告，并提供可被真实项目自动化流程调用的 gate 能力。
- 通过多个 Agent 自主推进变更。
- 将所有关键过程沉淀为可审查文件，而不是散落在聊天上下文中。

最终开发体验：

```text
/auok auto "新增工具调用评测能力，覆盖天气查询必须调用 weather 工具"
```

也可以分步执行：

```text
/auok init
/auok proposal "新增工具调用评测能力，覆盖天气查询必须调用 weather 工具"
/auok implement <change-id>
```

系统自动完成：

```text
当前主 Agent 作为 Orchestrator
-> 创建或复用 OpenSpec change
-> 通过 proposal 阶段拉起独立 Spec Agent 生成 proposal/design/tasks/spec delta
-> 拉起独立 Dev Agent 实现代码和 scenario
-> 拉起独立 QA Agent 运行 validate/run/grade/report/gate
-> 拉起独立 Review Agent 审查代码、规范、证据
-> 失败时 Orchestrator 自主返工协调
-> 拉起 Archive Agent 汇总证据和归档候选
-> 等待人工确认 archive / merge / release
```

auok Harness 使用的工具与自身职责：

| 组件 | 在 auok 中负责 | 不负责 |
| --- | --- | --- |
| OpenSpec 工具 | 需求、接口、行为规范、验收标准、变更和归档知识 | 执行任务、调度 Agent |
| Superpowers 工具 | 澄清、设计、计划、TDD、调试、评审、验证等工程纪律 | 保存工程事实、调度团队 |
| auok Harness | `/auok` 入口、Agent 分工、任务调度、权限边界、状态机、handoff、run/grade/report/gate | 替代规范工具做需求决策、绕过验证直接完成 |

当前 harness 工程与真实项目工作空间的映射：

| 组件 | 本仓库实现 / 模板源 | 真实项目运行时工作空间 |
| --- | --- |
| OpenSpec 工具 | `openspec/specs`、`openspec/changes` 作为 harness 规范和模板 | `auok/openspec/specs`、`auok/openspec/changes/<change-id>/proposal.md`、`design.md`、`tasks.md` |
| Superpowers 工具 | `superpowers/usage.md`、`AGENTS.md`、安装模板生成的 `~/.codex/skills/auok/SKILL.md`、`~/.claude/commands/auok.md` | Codex / Claude 当前会话按 `/auok` 指令约束执行 |
| auok Harness | `harness/` 源码、`agent-orchestration/workflows`、`contracts`、安装模板 | `auok/architecture`、`auok/orchestration/states`、`handoffs`、`workflows`、`auok/harness/scenarios`、`auok/runs` |

因此实现目标不是“auok 替 OpenSpec 写规范”或“Superpowers 调度 Agent”，而是：

```text
auok 使用 OpenSpec 产出标准
auok 使用 Superpowers 约束每个 Agent 怎么做
auok Harness 让 Orchestrator 拉起并管理多个独立 Agent 完成工作
```

## 3. 多 Agent 角色设计

### 3.1 Orchestrator Agent

职责：

- 接收目标。
- 创建或选择 workflow。
- 创建状态文件。
- 拉起独立 Spec / Dev / QA / Review / Archive 子 Agent。
- 分派任务给其他 Agent。
- 收集 handoff 和 evidence。
- 根据 gate 结果决定推进、返工或等待人工确认。
- 自主决定下一个 Agent 和重试路径。
- 维护最大重试次数和 blocked 状态。

不可做：

- 不直接修改业务实现。
- 不跳过失败 gate。
- 不自行降低验收标准。
- 不要求用户协调 Agent。
- 不把单 Agent 扮演多个角色当作 `/auok implement` 的默认实现。

输出：

```json
{
  "agent": "orchestrator",
  "change": "add-tool-call-eval",
  "state": "qa_failed",
  "next": "dev_fix",
  "reason": "auok gate failed: pass_rate=0.91 < 1.0"
}
```

### 3.2 Spec Agent

职责：

- 阅读 `openspec/specs/`。
- 创建 `openspec/changes/<change-id>/`。
- 生成 `proposal.md`、`design.md`、`tasks.md`、`specs/*/spec.md`。
- 明确不做什么、验收标准、风险和影响面。

Superpowers 使用重点：

- `brainstorming`
- `writing-plans`

输出：

```json
{
  "agent": "spec",
  "status": "done",
  "files": [
    "openspec/changes/add-tool-call-eval/proposal.md",
    "openspec/changes/add-tool-call-eval/design.md",
    "openspec/changes/add-tool-call-eval/tasks.md"
  ],
  "next": "dev"
}
```

### 3.3 Dev Agent

职责：

- 按 `tasks.md` 实现代码。
- 优先补测试或 scenario。
- 实现 adapter、runner、grader、report、CLI。
- 更新任务状态。

Superpowers 使用重点：

- `test-driven-development`
- `executing-plans`
- `systematic-debugging`

不可做：

- 不改需求。
- 不降低 gate。
- 不删除无关代码。

### 3.4 QA Agent

职责：

- 设计 scenario。
- 校验 scenario schema。
- 执行 `auok run`、`auok grade`、`auok gate`。
- 生成测试报告。
- 把失败转成可执行反馈。

输出：

```json
{
  "agent": "qa",
  "status": "failed",
  "evidence": [
    "runs/add-tool-call-eval/report.md",
    "auok gate failed: critical scenario tool-call.weather.basic failed"
  ],
  "next": "dev_fix"
}
```

### 3.5 Review Agent

职责：

- 审查 spec 是否覆盖需求。
- 审查代码是否符合 design/tasks。
- 审查测试是否能证明行为。
- 审查是否存在过度设计、未验证声明、无关改动。

Superpowers 使用重点：

- `requesting-code-review`
- `receiving-code-review`

输出要求：

- findings 优先，按严重程度排序。
- 每个问题必须指向文件和证据。
- 没有问题时明确说明残余风险。

### 3.6 Archive Agent

职责：

- 确认所有 gate 通过。
- 确认 tasks 完成。
- 准备 `openspec archive`。
- 生成归档候选报告。

不可做：

- 不在有失败 gate 时归档。
- 不自动合入主分支。
- 不自动发布。

## 4. 状态机设计

状态文件放在真实项目的 `auok/` 工作空间中：

```text
auok/orchestration/states/<change-id>.json
```

示例：

```yaml
change: add-tool-call-eval
goal: 新增工具调用评测能力，覆盖天气查询必须调用 weather 工具
state: implementing
owner: orchestrator

agents:
  spec:
    status: done
    last_output: auok/orchestration/handoffs/add-tool-call-eval/spec-to-dev.json
  dev:
    status: running
  qa:
    status: pending
  review:
    status: pending
  archive:
    status: pending

  gates:
    openspec_validate:
      status: pass
      command: auok validate add-tool-call-eval
  unit_tests:
    status: pending
    command: npm test
  auok_gate:
    status: pending
    command: auok gate runs/add-tool-call-eval --min-pass-rate 1.0 --no-critical-failures
  review:
    status: pending

human_approval:
  required_for:
    - archive
    - merge
    - release
    - lowering_gate_threshold
    - production_config_change
  status: not_requested
```

审批证据包必须包含：

- change id。
- proposal / design / tasks 路径。
- OpenSpec validate 结果。
- 单测或项目测试结果。
- `auok gate` 结果。
- Review Agent findings。
- Archive Agent 建议动作。

核心状态：

```text
created
-> spec_planning
-> spec_review
-> implementing
-> qa_running
-> qa_verified
-> review_running
-> ready_for_archive
-> archived
```

失败分支：

```text
qa_failed -> fixing
review_failed -> fixing
openspec_invalid -> spec_planning
blocked -> waiting_human_input
```

`blocked` 的进入条件：

- 自动重试超过上限。
- 需求或验收标准冲突。
- 需要降低 gate 阈值。
- 需要生产配置或敏感权限。
- Agent 无法从现有证据判断下一步。

## 5. Workflow 定义

Workflow 文件放在：

```text
agent-orchestration/workflows/change-lifecycle.yaml
```

示例：

```yaml
name: change-lifecycle
description: 多 Agent 自主完成 OpenSpec change 到归档候选

states:
  created:
    next: spec_planning

  spec_planning:
    agent: spec
    inputs:
      - openspec/specs
      - user_goal
    outputs:
      - openspec/changes/{change}/proposal.md
      - openspec/changes/{change}/design.md
      - openspec/changes/{change}/tasks.md
    next: spec_review

  spec_review:
    agent: review
    command: auok validate {change}
    on_pass: implementing
    on_fail: spec_planning

  implementing:
    agent: dev
    inputs:
      - openspec/changes/{change}/tasks.md
    next: qa_running

  qa_running:
    agent: qa
    commands:
      - auok run <scenario-id-or-capability-selected-by-qa-agent> --out runs/{change}
      - auok grade runs/{change}
      - auok gate runs/{change} --min-pass-rate 1.0 --no-critical-failures
    on_pass: qa_verified
    on_fail: fixing

  qa_verified:
    agent: review
    next: review_running

  review_running:
    agent: review
    on_pass: ready_for_archive
    on_fail: fixing

  fixing:
    agent: dev
    next: qa_running

  ready_for_archive:
    agent: archive
    waits_for: /auok archive <change-id>
```

### 5.1 多 Agent 运行时

多 Agent 的主入口是 Codex / Claude 当前会话里的 slash command。当前主 Agent 是 Orchestrator，它必须使用工具环境提供的子 Agent / task / subagent 能力拉起独立 Agent，而不是自己扮演所有角色：

```text
/auok proposal "需求"
/auok auto "需求"
/auok implement <change-id>
/auok archive <change-id>
```

安装 slash command 的命令：

```bash
npx github:ZongxingH/are-you-ok install --target codex --lang zh
npx github:ZongxingH/are-you-ok install --target claude --lang zh
npx github:ZongxingH/are-you-ok install --target all --lang zh
```

安装命令必须同时安装本地 runtime：

```text
~/.auok/runtime
```

Codex / Claude 的 `/auok` 指令只能调用该本地 runtime；不得在 `/auok init`、`/auok proposal`、`/auok auto`、`/auok implement` 执行时再临时下载远程代码。

安装语言：

```text
zh -> 中文交互指令，默认值
en -> 英文交互指令
```

生成：

```text
Codex  -> ~/.codex/skills/auok/SKILL.md
Claude -> ~/.claude/commands/auok.md
```

安装产物只暴露主入口，角色 Agent 是 `/auok` Orchestrator 的内部实现，不再安装为独立可见命令：

```text
auok
```

Codex/Claude 当前会话 Agent 读取这些 command 后，只作为 Orchestrator：

- 创建或恢复 change。
- 拉起独立 Dev Agent。
- 拉起独立 QA Agent。
- 拉起独立 Review Agent。
- 拉起独立 Archive Agent。
- 根据 handoff、state、gate 结果自主决定下一步。
- 失败时自主拉起 Dev/QA/Review 返工，直到通过或进入 blocked。

底层 Node 代码负责 `/auok` 工作流需要的确定性状态、文档、验证和 gate 能力；用户不直接使用终端 CLI 作为工作流入口。

必须保证：

- 每个 Agent 的输入输出文件明确。
- 状态推进可恢复。
- 失败可返工。
- 所有完成声明有命令证据。
- 当前主 Agent 不要求用户协调 Agent。
- 用户不需要手动调用 Spec、Dev、QA、Review、Archive，主 Agent 必须自主编排。
- `/auok auto` 从原始需求开始，按需初始化，再完成 proposal 和 implement，最后停在 `ready_for_archive`。
- `/auok implement` 只接续已有 proposal；如果 proposal 不存在或不完整，必须 blocked 并提示先执行 `/auok proposal` 或改用 `/auok auto`。
- Dev Agent 可以自主改代码，但 archive、merge、release、降低 gate、生产配置修改必须人工批准。
- 如果当前工具环境不支持独立子 Agent，`/auok auto` 和 `/auok implement` 必须明确报 blocked，而不是降级为单 Agent 多角色执行，除非用户显式允许。

## 6. OpenSpec 使用方式

OpenSpec 是所有 Agent 的共享事实库。

初始化由 `/auok init` 完成，底层确定性能力只创建真实项目 `auok/openspec` 的目录和模板；OpenSpec 的语义理解、proposal/design/tasks/spec delta 精修由 Spec Agent 通过 skill 完成。

推荐 specs：

```text
openspec/specs/harness-core/spec.md
openspec/specs/scenario-model/spec.md
openspec/specs/runner/spec.md
openspec/specs/adapter/spec.md
openspec/specs/grader/spec.md
openspec/specs/report/spec.md
openspec/specs/ci-gate/spec.md
openspec/specs/agent-orchestration/spec.md
```

每个变更：

```text
openspec/changes/<change-id>/
  proposal.md
  design.md
  tasks.md
  specs/
```

Agent 必跑校验：

```bash
npm run auok -- validate <change-id>
```

归档候选由 Archive Agent 准备；真正归档只在用户执行 `/auok archive <change-id>` 后发生。

## 7. Superpowers 使用方式

Superpowers 约束每个 Agent 的做事方式，不保存长期事实。auok 不再通过脚本单独安装或复制 Superpowers；它通过安装后的 auok role skill 把 Superpowers 的工程纪律固化到 Agent 行为中。

各 Agent 默认行为：

| Agent | 必用 Superpowers 能力 |
| --- | --- |
| Spec Agent | brainstorming、writing-plans |
| Dev Agent | test-driven-development、executing-plans、systematic-debugging |
| QA Agent | verification-before-completion |
| Review Agent | requesting-code-review、receiving-code-review |
| Archive Agent | verification-before-completion |
| Orchestrator | writing-plans、verification-before-completion |

关键规则：

- Superpowers 产生的结论必须写回 OpenSpec 或 handoff 文件。
- 不能只在聊天里完成设计。
- 不能用“我认为通过了”替代命令输出。
- 失败必须转成状态文件和下一步任务。

## 8. /auok 入口与确定性能力设计

`/auok` 是用户唯一的日常入口，运行在 Codex / Claude 当前会话中。底层 Node 代码只用于安装 slash command、初始化真实项目 `auok/` 工作空间、状态落盘、验证和调试，不作为用户日常入口。

确定性能力在 `package.json` 中声明 bin，供 `/auok` 指令内部或 harness 工程调试使用：

```json
{
  "name": "auok",
  "bin": {
    "auok": "harness/cli/main.js"
  }
}
```

本仓库调试该确定性能力时可以使用：

```bash
npm run auok -- --help
```

该能力可以调用 OpenSpec，也可以直接读写真实项目中的 `auok/openspec`、`auok/orchestration` 和 `auok/runs`。对用户来说，日常入口统一使用 `/auok`。

### 8.1 Slash Command 生命周期入口

```text
/auok init
/auok proposal "需求"
/auok auto "需求"
/auok implement <change-id>
/auok archive <change-id>
```

说明：生命周期语义参考 OpenSpec OPSX，但用户入口是 Codex / Claude 中的 `/auok` slash command，不再要求用户使用终端 CLI。

用户可见入口只保留五个流程命令：

1. `/auok init`：初始化真实项目的 `auok/` 工作空间和架构上下文。
2. `/auok proposal "需求"`：只完成提案阶段，创建或完善 OpenSpec change，不进入实现、QA、Review 或 Archive。
3. `/auok auto "需求"`：由当前主 Agent 作为 Orchestrator；按需初始化，完成提案和实现流程，最后停在 `ready_for_archive`。
4. `/auok implement <change-id>`：由当前主 Agent 作为 Orchestrator；只接续已有 proposal，不接收新提案内容，然后拉起独立 Dev/QA/Review/Archive Agent 完成实现、测试、评审和归档候选。
5. `/auok archive <change-id>`：作为用户最终确认，要求状态已到 `ready_for_archive`。

`/auok auto` 和 `/auok implement` 的边界必须严格区分：

- `/auok auto "需求"` 是从原始需求开始的全流程入口，可以按需 init、创建或复用 change、完成 proposal、继续实现并停在归档候选。
- `/auok implement <change-id>` 是已有 proposal 之后的实现入口，只能接收 change id，只能执行 Dev/QA/Review/Archive 候选准备。
- `/auok implement` 不得接收一句新需求，不得创建 change，不得调用 `proposal --goal`，不得重写 proposal/design/tasks/spec delta。
- 如果用户误把原始需求传给 `/auok implement`，Orchestrator 必须报告命令不匹配，并提示使用 `/auok auto "需求"` 或 `/auok proposal "需求"`，不能静默改走 auto。

入口语义：

| 命令 | 语义 |
| --- | --- |
| `/auok init` | 初始化真实项目的 `auok/` 工作空间；空目录只创建基础目录；存量项目先由底层确定性能力生成占位初稿，再由当前大模型通过内部 Architect Agent 通读项目证据并完善 `auok/architecture`；包含模块级技术栈线索，例如 Redis、MongoDB、MySQL、Nacos 等；文档语言跟随安装语言，默认中文 |
| `/auok proposal "需求"` | 创建或完善 `auok/openspec/changes/<change-id>` 下的 proposal、design、tasks 和 spec delta；状态推进到实现前，不启动 Dev/QA/Review/Archive |
| `/auok auto "需求"` | 自动完成 init、proposal 和 implement；如果工作空间不存在先初始化，然后创建或完善 proposal，再编排 Dev、QA、Review、Archive，最终停在 `ready_for_archive` 等待人工归档 |
| `/auok implement <change-id>` | proposal 完成后的自主实现入口；主 Agent 作为 Orchestrator，只接续已有 proposal，再编排 Dev、QA、Review、Archive 等独立子 Agent 完成需求开发 |
| `/auok archive <change-id>` | 用户最终确认归档；要求状态已到 `ready_for_archive` |

### 8.2 `/auok` 内部可调用的确定性命令

这些命令由 `/auok` command 内部或工具开发调试使用，不作为用户工作流入口：

```bash
npm run auok -- init
npm run auok -- new <change-id>
npm run auok -- proposal <change-id> --goal "需求"
npm run auok -- agent status <change-id>
npm run auok -- validate <change-id>
npm run auok -- verify <change-id>
npm run auok -- run ...
npm run auok -- grade ...
npm run auok -- report ...
npm run auok -- gate ...
npm run auok -- lifecycle transition <change-id> --to <state>
npm run auok -- lifecycle ready-for-archive <change-id>
npm run auok -- lifecycle block <change-id> --reason "..."
npm run auok -- agent handoff <change-id> --from <agent> --to <agent>
npm run auok -- archive <change-id>
npm run auok -- install --target all
```

命令语义：

| 命令 | 作用 |
| --- | --- |
| `new` | 在真实项目 `auok/` 工作空间中创建 OpenSpec change 的确定性文件骨架 |
| `proposal` | 创建 proposal 阶段骨架、初始化 spec-to-dev handoff，并把状态推进到 proposal 阶段 |
| `agent status` | 读取真实项目 `auok/orchestration/states` 下的状态，供 `/auok` Orchestrator 决策 |
| `validate/verify` | 校验真实项目 `auok/` 工作空间中的规范、场景、状态和 gate |
| `run/grade/report/gate` | 执行 harness 验证闭环 |
| `lifecycle transition` | 按允许状态机推进状态，不做智能判断 |
| `lifecycle ready-for-archive` | 在 gate、review、archive handoff 证据满足时将状态置为 `ready_for_archive` |
| `lifecycle block` | 将状态置为 `blocked` 并记录阻塞原因 |
| `agent handoff/status/resume` | 维护真实项目 `auok/orchestration` 下的 Agent 交接和状态 |
| `archive` | 在用户触发 `/auok archive` 后执行归档 |

`validate` 只保留按 change 校验的形式，不提供 `validate --all` 作为用户或内部工作流步骤。全局工作空间检查如有需要，应由 `/auok init` 和具体 change lifecycle 的确定性检查覆盖。

### 8.3 Lifecycle 状态推进

Lifecycle 命令只负责确定性状态读写和合法转移检查，不负责选择下一步、不负责判断需求正确性，也不替代 Orchestrator。

允许的主干转移：

```text
created
-> spec_planning
-> spec_review
-> implementing
-> qa_running
-> qa_verified
-> review_running
-> ready_for_archive
-> archived
```

允许的失败转移：

```text
openspec_invalid -> spec_planning
qa_failed -> fixing -> qa_running
review_failed -> fixing -> qa_running
blocked -> waiting_human_input
```

`ready_for_archive` 只能由 `/auok` 内部调用 `lifecycle ready-for-archive <change-id>` 设置。该命令必须至少检查：

- `openspec_validate.status` 为 `pass`。
- `auok_gate.status` 为 `pass`，且没有失败项。
- `review` 通过或存在 Review Agent 的通过 handoff。
- Archive Agent 已写出标准 `archive-to-human.json` handoff，包含归档候选摘要、证据、复现命令和下一步动作。
- 没有未解决的 critical failure 或 blocking finding。

### 8.4 评测执行命令

```bash
npm run auok -- list scenarios
npm run auok -- run --capability tool-calling --adapter mock --out tool-call-eval
npm run auok -- grade tool-call-eval
npm run auok -- report tool-call-eval
npm run auok -- compare baseline tool-call-eval
npm run auok -- gate tool-call-eval --min-pass-rate 1.0 --no-critical-failures
```

变更影响分析不放在底层确定性能力中硬编码。QA Agent 根据 OpenSpec change、handoff 和项目证据选择 scenario 或 capability，再调用 `run/grade/report/gate`。

### 8.5 JSON 输出

所有自动化命令必须支持 `--json`：

```json
{
  "change": "add-tool-call-eval",
  "state": "qa_failed",
  "agents": {
    "spec": "done",
    "dev": "done",
    "qa": "failed",
    "review": "pending"
```

## 9. Handoff 协议

Agent 之间不得只靠对话交接，必须写 handoff 文件。

路径：

```text
auok/orchestration/handoffs/<change-id>/<from>-to-<to>.json
```

格式：

```json
{
  "change": "add-tool-call-eval",
  "from": "qa",
  "to": "dev",
  "status": "failed",
  "summary": "critical scenario failed because no weather tool call was emitted",
  "evidence": [
    "runs/add-tool-call-eval/report.md",
    "runs/add-tool-call-eval/results.jsonl"
  ],
  "required_action": [
    "Inspect adapter output normalization",
    "Fix tool_calls extraction",
    "Re-run auok gate"
  ],
  "commands_to_reproduce": [
    "auok run tool-call.weather.basic --adapter mock --out runs/repro",
    "auok gate runs/repro --min-pass-rate 1.0 --no-critical-failures"
  ]
}
```

## 10. Harness 领域模型

### 10.1 Scenario

```yaml
id: tool-call.weather.basic
title: 查询天气时必须调用天气工具
capability: tool-calling
severity: critical
tags: [tool-call, weather, smoke]
input:
  user: "帮我查一下今天北京天气"
expected:
  tool_calls:
    - name: weather
grader:
  type: rule
  rules:
    - path: "$.tool_calls[*].name"
      contains: weather
```

### 10.2 Adapter

```ts
interface HarnessAdapter {
  name: string
  invoke(input: ScenarioInput, context: RunContext): Promise<AdapterResult>
}
```

内置 adapter 至少包括：

- `mock`：保证 harness 自身可测。
- `http`：调用业务 HTTP API 或模型 API。
- `cli`：通过命令行适配本地 Agent、模型包装器或脚本。

### 10.3 Grader

优先级：

1. `rule`：确定性评分，适合 hard gate。
2. `snapshot`：黄金样本对比，适合结构化报告。
3. `judge`：LLM-as-judge，适合语义质量，不默认作为 hard gate。

### 10.4 Report

```text
runs/<run-id>/
  run.yaml
  results.jsonl
  summary.json
  report.md
```

## 11. 人工介入边界

尽量减少人工介入，但保留必要关口。

Agent 可自动执行：

- 创建 OpenSpec change。
- 生成 proposal/design/tasks/spec delta。
- 实现代码。
- 生成 scenario。
- 跑测试和 gate。
- 审查代码。
- 修复失败。
- 生成交付报告。
- 准备归档候选。

必须人工确认：

- 最终 merge。
- 最终 archive。
- release。
- 降低 gate 阈值。
- 修改生产配置。
- 删除大量文件。
- 启动大规模付费模型评测。
- 接受带残余 critical 风险的变更。

原因：如果 Spec Agent 一开始理解错目标，后续 Agent 可能会高效地实现错误方案。保留最终确认是必要的风险控制。

## 12. 门禁标准

auok Harness 提供确定性的 `validate/run/grade/report/gate` 能力，供 `/auok` 工作流、目标项目脚本或目标项目 CI 调用。本 harness 工程不内置验证自身源码仓库的 CI/self-check 代码。

门禁标准：

- OpenSpec 结构合法。
- Scenario schema 合法。
- Smoke 场景全部通过。
- Critical 场景不得失败。
- 报告必须成功生成。
- Review Agent 无 blocking finding。

## 13. 全功能实现范围

本 harness 工程按全功能目标实现，不再划分 MVP 版本。实现范围包括：

### 13.1 入口与安装

- 安装 `/auok` 主入口；角色 Agent 作为 Orchestrator 内部编排角色，不安装为独立用户命令。
- `/auok` 用户可见流程只包含 init、proposal、auto、implement、archive。
- Codex / Claude 指令默认中文，可通过安装参数生成英文版本。
- 底层确定性能力仅供 `/auok` 内部调用或本仓库调试，不作为用户工作流入口。

### 13.2 真实项目工作空间

- `/auok init` 在真实项目中创建 `auok/` 工作空间。
- 初始化 `auok/architecture`、`auok/openspec`、`auok/orchestration`、`auok/harness`、`auok/runs`。
- 存量项目的架构分析由内部 Architect Agent 完成；底层确定性能力只生成占位文件和模板。

### 13.3 OpenSpec 与 proposal

- `/auok proposal` 创建或完善 `auok/openspec/changes/<change-id>`。
- Spec Agent 生成 proposal、design、tasks、spec delta、验收标准、风险和范围。
- proposal 阶段只产出规范和 handoff，不启动实现、QA、Review 或 Archive。
- `/auok auto` 会复用同一 proposal 阶段，但不会停在 proposal；它会继续进入实现流程。

### 13.4 多 Agent 编排

- `/auok auto` 由当前主 Agent 作为 Orchestrator，按需初始化工作空间，创建或完善 proposal，然后继续编排实现、QA、Review 和归档候选。
- `/auok implement` 由当前主 Agent 作为 Orchestrator。
- 在 `/auok implement` 中，Orchestrator 只接续已有 proposal；不创建、不重写、不接收新的 proposal 内容。
- Orchestrator 编排 Dev、QA、Review、Archive Agent。
- 如果环境不支持独立子 Agent，必须进入 blocked，除非用户显式允许降级。
- QA 或 Review 失败时，Orchestrator 根据 handoff 自主返工，直到通过或进入 blocked。

### 13.5 状态机与 handoff

- 实现 `auok/orchestration/states/<change-id>.json` 状态读写。
- 实现合法 lifecycle transition、ready-for-archive、block。
- 实现 `auok/orchestration/handoffs/<change-id>/<from>-to-<to>.json`。
- 每个 handoff 必须包含状态、摘要、证据、复现命令和下一步动作。
- 所有状态变化必须有文件记录和必要证据。

### 13.6 执行、评分、报告和 gate

- 实现 scenario schema 校验。
- 实现 mock、http、cli adapter。
- 实现 rule、snapshot、judge grader。
- 实现 run、grade、report、compare、gate。
- gate 必须支持 pass rate 阈值和 critical failure 阻断。
- 报告必须展示 pass/fail、分数、失败原因、critical failures 和趋势对比。

### 13.7 归档

- Archive Agent 汇总证据并准备归档候选，但必须落盘为标准 `archive-to-human.json` handoff，不使用 `archive-candidate.json` 等自创文件。
- `lifecycle ready-for-archive` 检查 `openspec_validate=pass`、`auok_gate=pass`、`review-to-archive`、`archive-to-human` 和 blocking finding。
- `/auok archive <change-id>` 需要用户最终确认。
- 不自动 merge、release、降低 gate 或修改生产配置。

验收标准：

- `/auok init` 可在真实项目中创建完整 `auok/` 工作空间。
- `/auok proposal "<goal>"` 可形成完整 OpenSpec change。
- `/auok auto "<goal>"` 可从原始需求自动推进到 `ready_for_archive`，只留下人工归档。
- `/auok implement <change-id>` 可推进到 `ready_for_archive`。
- QA 失败和 Review 失败能通过 handoff 返工。
- `ready_for_archive` 只能由 lifecycle 确定性检查设置。
- `/auok archive <change-id>` 只在用户确认且状态满足时归档。
- 所有完成声明都有命令证据和文件证据。

## 15. 成功标准

方案落地成功的标志：

- 一个目标可以通过 `/auok proposal "<goal>"` 形成完整提案。
- 一个目标可以通过 `/auok auto "<goal>"` 自动完成初始化、提案和实现，推进到归档候选。
- 一个已完成 proposal 的 change 可以通过 `/auok implement <change-id>` 自动推进到归档候选。
- OpenSpec change 完整，且能通过 validate。
- 多 Agent 之间通过状态文件和 handoff 文件交接。
- QA 失败和 Review 失败能自动返工。
- 所有完成声明都有命令证据。
- 真实项目可自行接入 `auok gate` 阻止关键评测失败合入。
- 人只在关键风险点介入。

## 16. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| Spec Agent 理解错需求 | proposal/design 进入人工轻审或 Review Agent 独立审查 |
| Agent 互相覆盖改动 | 每个 Agent 限定可写目录，状态机串行关键写操作 |
| Review Agent 自我确认 | Review Agent 独立上下文，基于 diff、spec 和测试证据审查 |
| LLM judge 波动 | 不作为默认 hard gate，除非 spec 明确要求且风险可接受 |
| 自动化误降门禁 | 降低阈值必须人工确认 |
| runs 数据膨胀 | 只提交 baseline 和摘要，大日志走 artifact |
| 循环修复无法收敛 | 设置最大重试次数，进入 `blocked` 等人工判断 |

建议最大自动重试：

```yaml
retry_policy:
  qa_failed:
    max_attempts: 3
  review_failed:
    max_attempts: 2
  openspec_invalid:
    max_attempts: 2
  on_exceeded: blocked
```

## 17. GitHub 分发与一键安装

后期上传 GitHub 时，建议不要提交 OpenSpec 和 Superpowers 的全局安装产物，也不要提交用户 HOME 目录下的 skill、plugin、symlink。正确方式是：**仓库提交 harness 工程源码、规范、模板和安装器；用户通过 `npx github:ZongxingH/are-you-ok install --target codex|claude|all --lang zh` 安装到本机 Agent 环境和 `~/.auok/runtime` 本地 runtime**。真实项目运行时产生的 `auok/` 工作空间属于使用方项目。

### 17.1 应该提交到 GitHub 的内容

```text
openspec/
  config.yaml
  specs/
  changes/

agent-orchestration/
  workflows/
  contracts/

superpowers/
  usage.md
  project-rules.md

AGENTS.md
package.json
README.md
```

```text
node_modules/
runs/* 大体量运行日志

本机 token / API key / .env
```
