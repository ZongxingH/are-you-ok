# auok Harness 工程方案

## 0. 方案结论

auok 是一个面向大模型编码环境的 Harness 工程实现。它通过 OpenSpec 和 Superpowers 两类工具，把“规范、纪律、编排、验证、归档”串成一个可执行闭环。

本方案不是把 OpenSpec、Superpowers、auok 当成三个并列产品，而是：

- **OpenSpec** 是 auok 使用的规范工具：定“做什么”。用结构化文档锁定需求、接口、行为规范、验收标准和归档后的 source of truth。
- **Superpowers** 是 auok 使用的工程纪律工具：定“怎么做”。把澄清、设计、计划、TDD、调试、评审、验证等工程流程固化为 Agent 行为约束。
- **auok Harness** 是最终实现：定“谁来做 + 怎么管”。负责 `/auok` 入口、角色分工、任务调度、权限边界、状态机、handoff、验证证据和最终闭环。

实现关系：

```text
auok Harness
-> 使用 OpenSpec 管规范和验收标准
-> 使用 Superpowers 管 Agent 工程纪律
-> 自己负责多 Agent 编排、验证、状态和归档闭环
```

`/auok` slash command、Multi-Agent Orchestrator、状态机、handoff、run/grade/report/gate 都属于 auok Harness 的实现组件。底层 Node 命令只作为内部 backend/CI/debug 工具，不作为用户日常入口。

目标不是让一个 Agent 从头猜到尾，而是让当前主 Agent 作为 Orchestrator 拉起多个独立 Agent，并基于共享文件和确定性命令协作。

推荐自动化边界：

- Agent 可以自动完成需求整理、OpenSpec change、设计、任务拆分、实现、测试、评审、修复、报告和归档候选。
- 人只保留关键审批：目标确认、风险边界确认、降低门禁阈值、生产配置修改、最终 merge / archive / release。

## 1. 建设目标

建设一个 AI/Agent harness 工程，用于：

- 管理评测能力规范。
- 管理测试场景和评测数据。
- 自动执行被测 Agent / 模型 / API。
- 自动评分、生成报告、执行 CI 门禁。
- 通过多个 Agent 自主推进变更。
- 将所有关键过程沉淀为可审查文件，而不是散落在聊天上下文中。

最终开发体验：

```text
/auok auto "新增工具调用评测能力，覆盖天气查询必须调用 weather 工具"
```

系统自动完成：

```text
当前主 Agent 作为 Orchestrator
-> 创建 OpenSpec change
-> 拉起独立 Spec Agent 生成 proposal/design/tasks/spec delta
-> 拉起独立 Dev Agent 实现代码和 scenario
-> 拉起独立 QA Agent 运行 validate/run/grade/report/gate
-> 拉起独立 Review Agent 审查代码、规范、证据
-> 失败时 Orchestrator 自主返工协调
-> 拉起 Archive Agent 汇总证据和归档候选
-> 等待人工确认 archive / merge / release
```

auok Harness 使用的工具与自身职责：

| 组件 | 在 auok 中负责 | 不负责 |
| --- | --- | --- |
| OpenSpec 工具 | 需求、接口、行为规范、验收标准、变更和归档知识 | 执行任务、调度 Agent |

| auok Harness | `/auok` 入口、Agent 分工、任务调度、权限边界、状态机、handoff、run/grade/report/gate | 替代规范工具做需求决策、绕过验证直接完成 |

当前实现映射：

| 组件 | 已实现载体 |
| --- | --- |
| OpenSpec 工具 | `auok/openspec/specs`、`auok/openspec/changes/<change-id>/proposal.md`、`design.md`、`tasks.md` |
| Superpowers 工具 | `superpowers/usage.md`、`AGENTS.md`、`.codex/commands/auok.md`、`.claude/commands/auok.md` 中的工程纪律规则 |
| auok Harness | `/auok` slash command、内部 backend、`auok/orchestration/states`、`handoffs`、`workflows`、`harness/scenarios`、`runs` |

因此实现目标不是“auok 替 OpenSpec 写规范”或“Superpowers 调度 Agent”，而是：

```text
auok 使用 OpenSpec 产出标准
auok 使用 Superpowers 约束每个 Agent 怎么做
auok Harness 让 Orchestrator 拉起并管理多个独立 Agent 完成工作
```

## 3. 多 Agent 角色设计

### 3.1 Orchestrator Agent

职责：

- 接收目标。
- 创建或选择 workflow。
- 创建状态文件。
- 拉起独立 Spec / Dev / QA / Review / Archive 子 Agent。
- 分派任务给其他 Agent。
- 收集 handoff 和 evidence。
- 根据 gate 结果决定推进、返工或等待人工确认。
- 自主决定下一个 Agent 和重试路径。
- 维护最大重试次数和 blocked 状态。

不可做：

- 不直接修改业务实现。
- 不跳过失败 gate。
- 不自行降低验收标准。
- 不要求用户协调 Agent。
- 不把单 Agent 扮演多个角色当作 `/auok auto` 的默认实现。

输出：

```json
{
  "agent": "orchestrator",
  "change": "add-tool-call-eval",
  "state": "qa_failed",
  "next": "dev_fix",
  "reason": "auok gate failed: pass_rate=0.91 < 1.0"
}
```

### 3.2 Spec Agent

职责：

- 阅读 `openspec/specs/`。
- 创建 `openspec/changes/<change-id>/`。
- 生成 `proposal.md`、`design.md`、`tasks.md`、`specs/*/spec.md`。
- 明确不做什么、验收标准、风险和影响面。

Superpowers 使用重点：

- `brainstorming`
- `writing-plans`

输出：

```json
{
  "agent": "spec",
  "status": "done",
  "files": [
    "openspec/changes/add-tool-call-eval/proposal.md",
    "openspec/changes/add-tool-call-eval/design.md",
    "openspec/changes/add-tool-call-eval/tasks.md"
  ],
  "next": "dev"
}
```

### 3.3 Dev Agent

职责：

- 按 `tasks.md` 实现代码。
- 优先补测试或 scenario。
- 实现 adapter、runner、grader、report、CLI。
- 更新任务状态。

Superpowers 使用重点：

- `test-driven-development`
- `executing-plans`
- `systematic-debugging`

不可做：

- 不改需求。
- 不降低 gate。
- 不删除无关代码。

### 3.4 QA Agent

职责：

- 设计 scenario。
- 校验 scenario schema。
- 执行 `auok run`、`auok grade`、`auok gate`。
- 生成测试报告。
- 把失败转成可执行反馈。

输出：

```json
{
  "agent": "qa",
  "status": "failed",
  "evidence": [
    "runs/add-tool-call-eval/report.md",
    "auok gate failed: critical scenario tool-call.weather.basic failed"
  ],
  "next": "dev_fix"
}
```

### 3.5 Review Agent

职责：

- 审查 spec 是否覆盖需求。
- 审查代码是否符合 design/tasks。
- 审查测试是否能证明行为。
- 审查是否存在过度设计、未验证声明、无关改动。

Superpowers 使用重点：

- `requesting-code-review`
- `receiving-code-review`

输出要求：

- findings 优先，按严重程度排序。
- 每个问题必须指向文件和证据。
- 没有问题时明确说明残余风险。

### 3.6 Archive Agent

职责：

- 确认所有 gate 通过。
- 确认 tasks 完成。
- 准备 `openspec archive`。
- 生成归档候选报告。

不可做：

- 不在有失败 gate 时归档。
- 不自动合入主分支。
- 不自动发布。

## 4. 状态机设计

状态文件放在：

```text
agent-orchestration/states/<change-id>.yaml
```

示例：

```yaml
change: add-tool-call-eval
goal: 新增工具调用评测能力，覆盖天气查询必须调用 weather 工具
state: implementing
owner: orchestrator

agents:
  spec:
    status: done
    last_output: agent-orchestration/handoffs/add-tool-call-eval/spec-to-dev.json
  dev:
    status: running
  qa:
    status: pending
  review:
    status: pending
  archive:
    status: pending

gates:
  openspec_validate:
    status: pass
    command: openspec validate add-tool-call-eval --strict
  unit_tests:
    status: pending
    command: npm test
  auok_gate:
    status: pending
    command: auok gate runs/add-tool-call-eval --min-pass-rate 1.0 --no-critical-failures
  review:
    status: pending

human_approval:
  required_for:
    - merge
    - archive
    - release
    - lowering_gate_threshold
    - production_config_change
  status: not_requested
```

`human_approval.status` 取值：

```text
not_requested  # 尚未请求人工确认
requested      # Archive Agent 已生成证据包，等待确认
approved       # 人已确认，可以继续 merge/release，或由 archive 命令自动记录归档确认
rejected       # 人拒绝，状态机回到 blocked 或 fixing
```

审批证据包必须包含：

- change id。
- proposal / design / tasks 路径。
- OpenSpec validate 结果。
- 单测或项目测试结果。
- `auok gate` 结果。
- Review Agent findings。
- Archive Agent 建议动作。

核心状态：

```text
created
-> spec_planning
-> spec_review
-> implementing
-> qa_running
-> review_running
-> fixing
-> ready_for_archive
-> waiting_human_approval
-> archived
```

失败分支：

```text
qa_failed -> fixing
review_failed -> fixing
openspec_invalid -> spec_planning
blocked -> waiting_human_input
```

`blocked` 的进入条件：

- 自动重试超过上限。
- 需求或验收标准冲突。
- 需要降低 gate 阈值。
- 需要生产配置或敏感权限。
- Agent 无法从现有证据判断下一步。

## 5. Workflow 定义

Workflow 文件放在：

```text
agent-orchestration/workflows/change-lifecycle.yaml
```

示例：

```yaml
name: change-lifecycle
description: 多 Agent 自主完成 OpenSpec change 到归档候选

states:
  created:
    next: spec_planning

  spec_planning:
    agent: spec
    inputs:
      - openspec/specs
      - user_goal
    outputs:
      - openspec/changes/{change}/proposal.md
      - openspec/changes/{change}/design.md
      - openspec/changes/{change}/tasks.md
    next: spec_review

  spec_review:
    agent: review
    command: openspec validate {change} --strict
    on_pass: implementing
    on_fail: spec_planning

  implementing:
    agent: dev
    inputs:
      - openspec/changes/{change}/tasks.md
    next: qa_running

  qa_running:
    agent: qa
    commands:
      - auok run --change {change} --out runs/{change}
      - auok grade runs/{change}
      - auok gate runs/{change} --min-pass-rate 1.0 --no-critical-failures
    on_pass: review_running
    on_fail: fixing

  review_running:
    agent: review
    on_pass: ready_for_archive
    on_fail: fixing

  fixing:
    agent: dev
    next: qa_running

  ready_for_archive:
    agent: archive
    requires_human_approval: true
    next: waiting_human_approval
```

### 5.1 多 Agent 运行时

多 Agent 的主入口是 Codex / Claude 当前会话里的 slash command。当前主 Agent 是 Orchestrator，它必须使用工具环境提供的子 Agent / task / subagent 能力拉起独立 Agent，而不是自己扮演所有角色：

```text
/auok auto "需求"
/auok proposal "需求"
/auok archive <change-id>
```

安装 slash command 的命令：

```bash
npx github:ZongxingH/are-you-ok install --target codex
npx github:ZongxingH/are-you-ok install --target claude
npx github:ZongxingH/are-you-ok install --target all
```

如果后续发布到 npm registry，也可以使用短命令：

```bash
npx auok install --target all
```

生成：

```text
.codex/commands/auok.md
.claude/commands/auok.md
```

Codex/Claude 当前会话 Agent 读取这些 command 后，只作为 Orchestrator：

- 创建或恢复 change。
- 拉起独立 Spec Agent。
- 拉起独立 Dev Agent。
- 拉起独立 QA Agent。
- 拉起独立 Review Agent。
- 拉起独立 Archive Agent。
- 根据 handoff、state、gate 结果自主决定下一步。
- 失败时自主拉起 Dev/QA/Review 返工，直到通过或进入 blocked。

`auok` backend 负责底层状态、文档、验证和 gate；用户不直接使用终端 CLI 作为工作流入口。

必须保证：

- 每个 Agent 的输入输出文件明确。
- 状态推进可恢复。
- 失败可返工。
- 所有完成声明有命令证据。
- 当前主 Agent 不要求用户协调 Agent。
- Dev Agent 可以自主改代码，但 archive、merge、release、降低 gate、生产配置修改必须人工批准。
- 如果当前工具环境不支持独立子 Agent，`/auok auto` 必须明确报 blocked，而不是降级为单 Agent 多角色执行，除非用户显式允许。

## 6. OpenSpec 使用方式

OpenSpec 是所有 Agent 的共享事实库。

初始化：

```bash
npm install -g @fission-ai/openspec@<pinned-version>
openspec init --tools codex
```

如果项目已经把 OpenSpec 固定在 `devDependencies`，优先使用：

```bash
npx openspec init --tools codex
```

推荐 specs：

```text
openspec/specs/harness-core/spec.md
openspec/specs/scenario-model/spec.md
openspec/specs/runner/spec.md
openspec/specs/adapter/spec.md
openspec/specs/grader/spec.md
openspec/specs/report/spec.md
openspec/specs/ci-gate/spec.md
openspec/specs/agent-orchestration/spec.md
```

每个变更：

```text
openspec/changes/<change-id>/
  proposal.md
  design.md
  tasks.md
  specs/
```

Agent 必跑命令：

```bash
openspec status --change <change-id> --json
openspec validate <change-id> --strict
openspec instructions apply --change <change-id> --json
```

归档候选：

```bash
openspec archive <change-id> --yes
```

注意：`archive` 建议由 Archive Agent 准备，人确认后执行。

## 7. Superpowers 使用方式

Superpowers 约束每个 Agent 的做事方式，不保存长期事实。

安装到 Codex：

```text
Fetch and follow instructions from https://raw.githubusercontent.com/obra/superpowers/refs/heads/main/.codex/INSTALL.md
```

各 Agent 默认行为：

| Agent | 必用 Superpowers 能力 |
| --- | --- |
| Spec Agent | brainstorming、writing-plans |
| Dev Agent | test-driven-development、executing-plans、systematic-debugging |
| QA Agent | verification-before-completion |
| Review Agent | requesting-code-review、receiving-code-review |
| Archive Agent | verification-before-completion |
| Orchestrator | writing-plans、verification-before-completion |

关键规则：

- Superpowers 产生的结论必须写回 OpenSpec 或 handoff 文件。
- 不能只在聊天里完成设计。
- 不能用“我认为通过了”替代命令输出。
- 失败必须转成状态文件和下一步任务。

## 8. /auok 入口与内部 backend 设计

`/auok` 是用户唯一的日常入口，运行在 Codex / Claude 当前会话中。底层 Node backend 只用于 slash command 安装、状态落盘、验证、CI 和调试。

内部 backend 在 `package.json` 中声明 bin：

```json
{
  "name": "auok",
  "bin": {
    "auok": "harness/cli/main.js"
  }
}
```

工具仓库本地调试使用：

```bash
npm link
auok --help
```

CI 或内部调试时使用项目脚本转发：

```bash
npm run auok -- validate --all
```

backend 可以调用 OpenSpec，也可以直接读写 `auok/openspec`、`auok/orchestration` 和 `auok/runs`。对用户来说，日常入口统一使用 `/auok`。

### 8.1 Slash Command 生命周期入口

```text
/auok init
/auok proposal "需求"
/auok auto "需求"
/auok status <change-id> --json
/auok archive <change-id>
```

说明：生命周期语义参考 OpenSpec OPSX，但用户入口是 Codex / Claude 中的 `/auok` slash command，不再要求用户使用终端 CLI。

用户可见流程只有三阶段：

1. 提案：`/auok proposal "需求"` 生成或完善 OpenSpec change。
2. 自主编排 Agent 实现：`/auok auto "需求"` 由当前主 Agent 作为 Orchestrator，拉起独立 Spec/Dev/QA/Review/Archive Agent 完成实现、测试、评审和归档候选。
3. 归档：`/auok archive <change-id>` 作为用户最终确认，要求状态已到 `ready_for_archive`。

入口语义：

| 命令 | 语义 |
| --- | --- |
| `/auok init` | 初始化业务仓库的 `auok/` 工作区 |
| `/auok proposal "需求"` | 只完成提案阶段，生成或完善 OpenSpec proposal/design/tasks/spec delta |
| `/auok auto "需求"` | 主 Agent 作为 Orchestrator，拉起多个独立子 Agent 完成需求 |
| `/auok status <change-id>` | 输出 change、agent、gate 状态 |
| `/auok archive <change-id>` | 用户最终确认归档；要求状态已到 `ready_for_archive` |

### 8.2 内部 backend 命令

这些命令由 `/auok` command 内部、CI 或工具开发调试使用，不作为用户工作流入口：

```bash
npm run auok -- init
npm run auok -- new <change-id>
npm run auok -- ff <change-id>
npm run auok -- validate --all
npm run auok -- verify <change-id>
npm run auok -- run ...
npm run auok -- grade ...
npm run auok -- report ...
npm run auok -- gate ...
npm run auok -- archive <change-id>
npm run auok -- install --target all
```

命令语义：

| 命令 | 作用 |
| --- | --- |
| backend `new/ff` | 创建和准备 OpenSpec change |
| backend `validate/verify` | 校验规范、场景、状态和 gate |
| backend `run/grade/report/gate` | 执行 harness 验证闭环 |
| backend `agent handoff/status/resume` | 维护 Agent 交接和状态 |
| backend `archive` | 在用户触发 `/auok archive` 后执行归档 |

### 8.3 评测执行命令

```bash
npm run auok -- list scenarios
npm run auok -- run --capability tool-calling --adapter mock --out tool-call-eval
npm run auok -- grade tool-call-eval
npm run auok -- report tool-call-eval
npm run auok -- compare baseline tool-call-eval
npm run auok -- gate tool-call-eval --min-pass-rate 1.0 --no-critical-failures
```

`auok run --change <change-id>` 是自动化快捷入口，语义是：

- 读取 `openspec/changes/<change-id>/tasks.md` 和 spec delta。
- 找到本次变更新增或受影响的 scenarios。
- 如果没有可推导场景，则回退到 `--capability smoke`。
- 输出到 `runs/<change-id>/`。

第一阶段如果不实现变更影响分析，可以先要求显式传入 `--capability` 或 scenario id。

### 8.4 JSON 输出

所有自动化命令必须支持 `--json`：

```json
{
  "change": "add-tool-call-eval",
  "state": "qa_failed",
  "agents": {
    "spec": "done",
    "dev": "done",
    "qa": "failed",
    "review": "pending"
  },
  "gates": {
    "openspec_validate": "pass",
    "auok_gate": "fail"
  },
  "next": {
    "agent": "dev",
    "action": "fix failing critical scenario",
    "evidence": "runs/add-tool-call-eval/report.md"
  }
}
```

## 9. Handoff 协议

Agent 之间不得只靠对话交接，必须写 handoff 文件。

路径：

```text
agent-orchestration/handoffs/<change-id>/<from>-to-<to>.json
```

格式：

```json
{
  "change": "add-tool-call-eval",
```

## 10. Harness 领域模型

### 10.1 Scenario

```yaml
id: tool-call.weather.basic
title: 查询天气时必须调用天气工具
capability: tool-calling
severity: critical
tags: [tool-call, weather, smoke]
input:
  user: "帮我查一下今天北京天气"
expected:
  tool_calls:
    - name: weather
grader:
  type: rule
  rules:
    - path: "$.tool_calls[*].name"
      contains: weather
```

### 10.2 Adapter

```ts
interface HarnessAdapter {
  name: string
  invoke(input: ScenarioInput, context: RunContext): Promise<AdapterResult>
}
```

第一阶段只做：

- `mock`：保证 harness 自身可测。
- `api`：调用业务 API 或模型 API。

### 10.3 Grader

优先级：

1. `rule`：确定性评分，适合 hard gate。
2. `snapshot`：黄金样本对比，适合结构化报告。
3. `judge`：LLM-as-judge，适合语义质量，不默认作为 hard gate。

### 10.4 Report

```text
runs/<run-id>/
  run.yaml
  results.jsonl
  summary.json
  report.md
```

## 11. 人工介入边界

尽量减少人工介入，但保留必要关口。

Agent 可自动执行：

- 创建 OpenSpec change。
- 生成 proposal/design/tasks/spec delta。
- 实现代码。
- 生成 scenario。
- 跑测试和 gate。
- 审查代码。
- 修复失败。
- 生成交付报告。
- 准备归档候选。

必须人工确认：

- 最终 merge。
- 最终 archive。
- release。
- 降低 gate 阈值。
- 修改生产配置。
- 删除大量文件。
- 启动大规模付费模型评测。
- 接受带残余 critical 风险的变更。

原因：如果 Spec Agent 一开始理解错目标，后续 Agent 可能会高效地实现错误方案。保留最终确认是必要的风险控制。

## 12. CI 门禁

GitHub Actions 示例：

```yaml
name: harness

on:
  pull_request:
  push:
    branches: [main]

jobs:
  harness:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm install
      - run: npm run auok -- validate --all
      - run: npm run auok -- run --capability smoke --adapter mock --out runs/ci
      - run: npm run auok -- grade runs/ci
      - run: npm run auok -- report runs/ci --format md,json
      - run: npm run auok -- gate runs/ci --min-pass-rate 1.0 --no-critical-failures
```

门禁标准：

- OpenSpec 结构合法。
- Scenario schema 合法。
- Smoke 场景全部通过。
- Critical 场景不得失败。
- 报告必须成功生成。
- Review Agent 无 blocking finding。

## 13. MVP 实施计划

### 阶段 1：规范与规则底座

任务：

- 安装 OpenSpec。
- 安装 Superpowers。
- 初始化 `openspec/`。
- 创建初始 specs。
- 创建 `AGENTS.md`。
- 创建 Agent contracts。

验收：

- `openspec validate --all --strict` 通过。
- 每个 Agent 的输入、输出、可写范围明确。

### 阶段 2：auok backend 闭环

任务：

- 实现 scenario schema。
- 实现 `auok run`。
- 实现 mock adapter。
- 实现 rule grader。
- 实现 report。
- 实现 gate。

验收：

- 单个 scenario 可执行。
- `results.jsonl`、`summary.json`、`report.md` 可生成。
- gate 失败返回非 0。

### 阶段 3：多 Agent 状态机

任务：

- 实现 `agent-orchestration/workflows/change-lifecycle.yaml`。
- 实现状态文件读写。
- 实现 handoff 协议。
- 实现 `auok auto`、`auok agent status`、`auok agent resume`。
- 接入 Spec/Dev/QA/Review/Archive Agent。

验收：

- 一个简单 change 能自动走到 `ready_for_archive`。
- QA 失败能自动回到 Dev 修复。
- Review 失败能自动回到 Dev 修复。
- 所有状态变化有文件记录。

### 阶段 4：真实适配器与高级评分

任务：

- 接入真实 API adapter。
- 增加 snapshot grader。
- 增加 judge grader。
- 增加 baseline compare。
- 增加 CI artifact。

验收：

- 一个真实 capability 可稳定回归。
- 报告能展示 pass/fail、分数、失败原因、趋势对比。

## 14. 推荐 Prompt 模板

启动自动变更：

```text
使用 OpenSpec + Superpowers + 多 Agent 自主协作流程。
目标：新增工具调用评测能力，覆盖天气查询必须调用 weather 工具。
请由 Orchestrator 创建 change 状态机，分派 Spec、Dev、QA、Review、Archive Agent。
所有 Agent 必须通过 OpenSpec、状态文件和 handoff 文件交接。
完成前必须运行 openspec validate、auok run、auok gate。
最终只准备 archive 候选，不自动 merge 或 release。
```

恢复失败变更：

```text
恢复 change add-tool-call-eval。
先读取 agent-orchestration/states/add-tool-call-eval.yaml。
根据当前 state 和 gates 继续推进。
如果 QA 或 Review 失败，生成 handoff 给 Dev Agent 修复。
不要跳过失败 gate。
```

人工确认归档：

```text
检查 change add-tool-call-eval 是否满足归档条件。
列出 openspec validate、unit tests、auok gate、review 的证据。
如果全部通过，准备 archive 命令和归档摘要，等待我确认。
```

## 15. 成功标准

方案落地成功的标志：

- 一个目标可以通过 `auok auto "<goal>"` 自动推进到归档候选。
- OpenSpec change 完整，且能通过 validate。
- 多 Agent 之间通过状态文件和 handoff 文件交接。
- QA 失败和 Review 失败能自动返工。
- 所有完成声明都有命令证据。
- CI 能阻止关键评测失败合入。
- 人只在关键风险点介入。

## 16. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| Spec Agent 理解错需求 | proposal/design 进入人工轻审或 Review Agent 独立审查 |
| Agent 互相覆盖改动 | 每个 Agent 限定可写目录，状态机串行关键写操作 |
| Review Agent 自我确认 | Review Agent 独立上下文，基于 diff、spec 和测试证据审查 |
| LLM judge 波动 | 不作为第一阶段 hard gate |
| 自动化误降门禁 | 降低阈值必须人工确认 |
| runs 数据膨胀 | 只提交 baseline 和摘要，大日志走 artifact |
| 循环修复无法收敛 | 设置最大重试次数，进入 `blocked` 等人工判断 |

建议最大自动重试：

```yaml
retry_policy:
  qa_failed:
    max_attempts: 3
  review_failed:
    max_attempts: 2
  openspec_invalid:
    max_attempts: 2
  on_exceeded: blocked
```

## 17. GitHub 分发与一键安装

后期上传 GitHub 时，建议不要提交 OpenSpec 和 Superpowers 的全局安装产物，也不要提交用户 HOME 目录下的 skill、plugin、symlink。正确方式是：**仓库提交项目配置和 bootstrap 脚本，用户 clone 后一条命令完成安装和初始化**。

### 17.1 应该提交到 GitHub 的内容

```text
scripts/
  bootstrap.sh
  verify-env.sh
  install-openspec.sh
  install-superpowers-codex.sh
  install-superpowers-claude.sh

openspec/
  config.yaml
  specs/
  changes/

agent-orchestration/
  workflows/
  contracts/

superpowers/
  usage.md
  project-rules.md

AGENTS.md
package.json
README.md
```

说明：

- `openspec/specs` 和 `openspec/changes` 是项目事实，应该提交。
- `agent-orchestration/workflows` 和 `contracts` 是多 Agent 协作协议，应该提交。
- `superpowers/usage.md` 只保存本项目如何使用 Superpowers，不复制官方 Superpowers skill 内容。
- `scripts/bootstrap.sh` 负责本机安装和初始化。

### 17.2 不应该提交的内容

```text
node_modules/
runs/* 大体量运行日志
~/.codex/superpowers/
~/.agents/skills/superpowers
全局 npm 安装目录
本机 token / API key / .env
```

这些属于本机环境或敏感数据，应该通过 `.gitignore` 排除。

### 17.3 推荐 package.json scripts

```json
{
  "name": "auok",
  "version": "0.1.0",
  "bin": {
    "auok": "harness/cli/main.js"
  },
  "scripts": {
    "auok": "node harness/cli/main.js",
    "bootstrap": "bash scripts/bootstrap.sh",
    "verify-env": "bash scripts/verify-env.sh",
    "openspec-validate": "openspec validate --all --strict",
    "auok-smoke": "npm run auok -- run --capability smoke --adapter mock --out runs/smoke && npm run auok -- grade runs/smoke && npm run auok -- report runs/smoke && npm run auok -- gate runs/smoke --min-pass-rate 1.0 --no-critical-failures",
    "agent-status": "npm run auok -- agent status"
  },
  "devDependencies": {}
}
```

版本建议：

- MVP 阶段 `auok` 使用零外部依赖，保证无网络也能运行 smoke gate。
- OpenSpec CLI 作为可选外部工具安装；如果后续加入 npm 依赖，发布到 GitHub 前必须固定版本。
- 用 lockfile 保证 CI 和开发机安装一致。
- 全局安装仍可作为开发者便利项，但不要依赖全局安装作为唯一入口。

### 17.4 bootstrap.sh 设计

```bash
#!/usr/bin/env bash
set -euo pipefail

ROOT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
cd "$ROOT_DIR"

echo "[1/5] Checking Node.js..."
node --version
npm --version

echo "[2/5] Installing npm dependencies..."
npm install

echo "[3/5] Verifying OpenSpec availability..."
if command -v openspec >/dev/null 2>&1; then
  openspec --version
else
  echo "OpenSpec CLI not found. Install it separately when OpenSpec validation is required."
fi

echo "[4/5] Configuring Agent environment..."
case "${AUOK_AGENT_TARGET:-none}" in
  codex)
    bash scripts/install-superpowers-codex.sh
    ;;
  claude)
    bash scripts/install-superpowers-claude.sh
    ;;
  none)
    echo "Skipping Agent-specific Superpowers install."
    ;;
  *)
    echo "Unsupported AUOK_AGENT_TARGET: ${AUOK_AGENT_TARGET}"
    exit 1
    ;;
esac

echo "[5/5] Verifying project..."
npm link || echo "Skipping global auok link; use: npm run auok -- <command>"
npm run verify-env

echo "Bootstrap complete."
```

注意：

- `openspec/` 已作为项目事实提交到仓库，bootstrap 不自动覆盖。
- OpenSpec CLI 缺失时不阻塞本地 auok smoke 验证。
- Superpowers 安装会写用户 HOME 目录，所以必须由脚本在本机执行，不提交结果。
- bootstrap 支持 `--codex`、`--claude`、`--no-agent`，也支持 `AUOK_AGENT_TARGET=codex|claude|none`。

### 17.5 Agent 环境安装脚本

Codex：

```bash
#!/usr/bin/env bash
set -euo pipefail

SUPERPOWERS_DIR="${HOME}/.codex/superpowers"
SKILLS_LINK="${HOME}/.agents/skills/superpowers"

mkdir -p "${HOME}/.codex" "${HOME}/.agents/skills"

if [ ! -d "$SUPERPOWERS_DIR/.git" ]; then
  git clone https://github.com/obra/superpowers.git "$SUPERPOWERS_DIR"
else
  git -C "$SUPERPOWERS_DIR" pull --ff-only
fi

if [ ! -e "$SKILLS_LINK" ]; then
  ln -s "$SUPERPOWERS_DIR/skills" "$SKILLS_LINK"
fi

echo "Superpowers installed for Codex."
echo "Restart Codex so skills are discovered."
```

Claude：

```bash
#!/usr/bin/env bash
set -euo pipefail

SUPERPOWERS_DIR="${HOME}/.claude/superpowers"

mkdir -p "${HOME}/.claude"

if [ ! -d "$SUPERPOWERS_DIR/.git" ]; then
  git clone https://github.com/obra/superpowers.git "$SUPERPOWERS_DIR"
else
  git -C "$SUPERPOWERS_DIR" pull --ff-only
fi

cat > "${HOME}/.claude/AUOK_SUPERPOWERS.md" <<EOF
# auok Superpowers Notes

Superpowers has been cloned to:

${SUPERPOWERS_DIR}

Claude Code integration varies by local setup. Use the project's AGENTS.md,
superpowers/usage.md, and the cloned Superpowers materials as behavior guidance.
EOF
```

如果团队不希望脚本直接联网安装 Superpowers，可以使用：

```bash
npm run bootstrap -- --no-agent
AUOK_AGENT_TARGET=none npm run bootstrap
```

### 17.8 CI 中的安装方式

CI 不需要安装 Superpowers，因为 CI 只做确定性验证，不跑交互式 Agent。

```yaml
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
  with:
    node-version: 20
- run: npm install
- run: npm run auok -- validate --all
- run: npm run auok-smoke
```

也就是说：

- 开发者本机：`bootstrap` 安装项目依赖，并按选择配置 Codex、Claude 或不配置 Agent 环境。
- CI：只安装 npm 依赖，跑 OpenSpec validate 和 auok gate。
- Agent 环境：Codex 安装后重启 Codex；Claude 安装后按本机 Claude Code 约定读取 `AGENTS.md` 和 `superpowers/usage.md`。
